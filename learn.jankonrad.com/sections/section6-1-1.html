6.1.2 Backend, server, API a virtuální prostředí – jak připravit projekt na webovou část

V minulé části vznikla databáze reservations.db a konzolový program, který do ní umí zapisovat rezervace. Teď je potřeba pochopit, jak se z lokálního Python skriptu stane backend, proč je potřeba server, co je API, proč budeme používat FastAPI, proč nestačí jen spustit python.exe a jakou roli v tom všem hraje virtuální prostředí. V další lekci už na tuto teorii naváže skutečný backend v Pythonu.

Co se v této lekci naučíš

  • Co je frontend, backend, server a databáze.
  • Co je API a proč webová stránka běžně nepracuje přímo s databází.
  • Proč je pro tento projekt vhodné FastAPI.
  • Že FastAPI není součást základní instalace Pythonu a musí se doinstalovat.
  • Co je virtuální prostředí, jak funguje a proč patří ke konkrétnímu projektu.
  • Jak se ve Windows vytváří a aktivuje .venv.
  • Jak číst zápis @app.get("/") opravdu krok po kroku.
  • Proč samotné spuštění python.exe backend nerozběhne.
  • Jak vypadá nejmenší možný backend ve FastAPI.
  • Jak bude propojená konzole, databáze, backend a webová stránka v našem rezervačním systému.
Cíl kapitoly:
Přesně pochopit, kde backend v projektu sedí, proč vzniká, jak komunikuje s databází a proč potřebuje vlastní balíčky a vlastní prostředí.

Na co tato kapitola navazuje

  1. Existuje databáze reservations.db.
  2. Existuje Python program, který do této databáze zapisuje rezervace přes konzoli.
  3. Další krok je zobrazit stejná data i na webové stránce.
  4. Webová stránka ale nebude pracovat s databází přímo.
  5. Mezi stránku a databázi proto vstoupí backend, který poběží jako server.
Jedna věta, kterou si zapamatuj:
Databáze data ukládá, backend je čte nebo zapisuje a webová stránka je zobrazuje.
Krok 1: kde v projektu sedí frontend, backend, server a databáze

Webová aplikace se obvykle neskládá z jediné části. Každá část má jiný úkol a teprve dohromady tvoří celý systém.

  • Frontend = to, co se zobrazuje v prohlížeči (HTML, CSS, JavaScript).
  • Backend = Python program, který zpracovává logiku aplikace.
  • Databáze = místo, kde jsou uložená data.
  • Server = běžící program, který čeká na požadavky a vrací odpovědi.
V našem projektu:
Frontend = stránka s výpisem rezervací
Backend = Python program využívající FastAPI
Databáze = reservations.db
Co znamenají tyto části jednoduše
  • Frontend je viditelná část aplikace.
  • Backend rozhoduje, jaká data se načtou, uloží nebo vrátí.
  • Databáze slouží jako uložiště.
  • Server je způsob, jak backend běží a komunikuje přes web.
Krok 2: co je API a proč ho projekt potřebuje

API je rozhraní, přes které spolu komunikují programy. V tomto projektu to znamená, že webová stránka pošle backendu požadavek a backend vrátí odpověď.

  • Stránka požádá o data.
  • Backend otevře databázi nebo zpracuje požadovanou akci.
  • Backend vrátí výsledek.
  • Stránka výsledek zobrazí.
Důležité:
Prohlížeč běžně nepracuje přímo se souborem SQLite na disku. Proto je mezi webovou stránkou a databází potřeba backend.
Proč webová stránka nepůjde přímo do databáze
  • SQLite databáze je soubor na disku.
  • Prohlížeč neumí používat Python modul sqlite3.
  • Práce s databází musí být řízená backendem.
  • Backend tedy slouží jako prostředník mezi stránkou a databází.
Krok 3: proč budeme používat právě FastAPI

V Pythonu existuje více frameworků pro tvorbu backendu. Pro tento projekt budeme používat FastAPI, protože je přehledné, krátké a dobře se na něm vysvětluje princip webového backendu.

  • Kód je stručný a dobře čitelný.
  • Rychle vznikne první funkční server.
  • Snadno se v něm vysvětlují cesty jako / nebo /reservations.
  • Dobře zapadá do Python projektu se SQLite databází.
  • Je vhodné pro malé ukázky i větší aplikace.
Nejdůležitější princip:
URL adresa → Python funkce → odpověď serveru
Krok 4: FastAPI není součást Pythonu – musíme ho doinstalovat

V této části projektu se poprvé objevuje něco, co základní Python sám o sobě neobsahuje.

  • sqlite3 = modul, který bývá součástí standardní instalace Pythonu.
  • FastAPI = externí balíček, který je potřeba do projektu doinstalovat.
Důležité:
Řádek from fastapi import FastAPI bude fungovat až tehdy, když je FastAPI v daném prostředí skutečně nainstalované.
Balíček, modul a import
  • Balíček = něco, co se instaluje přes pip.
  • Modul = něco, co se pak v Pythonu načítá přes import.
  • pip install ... = instalace do prostředí.
  • import ... = použití v kódu.
Krok 5: virtuální prostředí – proč existuje a co dělá

Virtuální prostředí je oddělené prostředí pro konkrétní Python projekt. Obsahuje vlastní sadu nainstalovaných balíčků a je navázané právě na danou složku projektu.

  • Každý projekt může mít své vlastní balíčky.
  • Různé projekty si nepřepisují verze knihoven navzájem.
  • Balíčky instalované do jednoho projektu se automaticky nepletou do jiného.
  • Backend tak přesně ví, z jakého prostředí má brát své závislosti.
Spojitost s projektem:
Virtuální prostředí patří ke konkrétní složce projektu. Když se v této složce aktivuje, Python i terminál používají balíčky nainstalované právě pro tento projekt.
Co se ve virtuálním prostředí děje technicky
  • Vytvoří se speciální složka, často pojmenovaná .venv.
  • Uvnitř je vlastní Python interpreter a vlastní místo pro balíčky.
  • Na Windows obsahuje podsložku Scripts, kde jsou spouštěcí skripty a nástroje.
  • Balíčky instalované do tohoto prostředí se ukládají jen k tomuto projektu.
  • Když později projekt smažeš nebo přeneseš, můžeš prostředí jednoduše vytvořit znovu.
Proč je virtuální prostředí důležité právě pro backend
  • Backend obvykle používá externí balíčky, například FastAPI.
  • Tyto balíčky je potřeba mít nainstalované správně a předvídatelně.
  • Virtuální prostředí zajišťuje, že backend běží nad přesně těmi balíčky, které tento projekt potřebuje.
  • Když se projekt později přesune na jiný počítač nebo server, je jasné, jaké závislosti potřebuje.
Krok 6: vytvoření prostředí a instalace FastAPI ve Windows PowerShellu

Ve Windows PowerShellu je bezpečné držet se příkazů, které odpovídají skutečné Windows syntaxi.

Windows PowerShell – doporučený postup 
py -m venv .venv
.\.venv\Scripts\Activate.ps1
py -m pip install "fastapi[standard]"
Vysvětlení příkazů „slovo od slova“
  • py = spouštěč Pythonu ve Windows.
  • -m venv = spusť modul venv a vytvoř virtuální prostředí.
  • .venv = název složky, ve které se prostředí vytvoří.
  • .\.venv\Scripts\Activate.ps1 = aktivuj prostředí v PowerShellu.
  • py -m pip install "fastapi[standard]" = nainstaluj FastAPI a související nástroje.
Poznámka k Windows:
Na některých instalacích funguje spolehlivě py, zatímco příkaz python nemusí být správně dostupný. Proto jsou zde Windows příklady zapsané přes py.
Častá chyba: proč zde nepoužívat source .venv/bin/activate
  • source .venv/bin/activate patří do bash nebo zsh na Linuxu a macOS.
  • Windows PowerShell má jinou syntaxi a jinou cestu ke skriptům.
  • Ve Windows se používá složka Scripts, ne bin.
  • V PowerShellu se proto používá .\.venv\Scripts\Activate.ps1.
Krok 7: když aktivace z nějakého důvodu nefunguje

Virtuální prostředí není nutné vždy aktivovat. Přímo z něj lze spouštět Python i nainstalované nástroje přes jejich úplnou cestu.

Windows PowerShell – přímé spuštění bez aktivace 
.\.venv\Scripts\python.exe -m pip install "fastapi[standard]"
.\.venv\Scripts\python.exe -m uvicorn main:app --reload
Proč to funguje
  • Spouští se přímo Python interpreter z vytvořeného prostředí.
  • V tom interpreteru se pak spustí modul pip nebo uvicorn.
  • Tím se obejde problém s PATH i s aktivací.
  • Je to užitečné hlavně tehdy, když ve Windows nefunguje python nebo se nedaří aktivace přes PowerShell.
Prakticky:
Aktivace prostředí je pohodlná, ale pro samotné použití prostředí není nezbytná.
Krok 8: první minimální soubor backendu

Aby server věděl, co má obsluhovat, potřebuje soubor s definicí aplikace. V našem projektu to bude main.py.

main.py 
from fastapi import FastAPI

app = FastAPI()

@app.get("/")
def read_root():
    return {"message": "Ahoj ze serveru"}
Vysvětlení kódu „slovo od slova“
  • from fastapi import FastAPI = importujeme třídu FastAPI z nainstalovaného balíčku.
  • app = FastAPI() = vytvoříme objekt aplikace.
  • @app.get("/") = když přijde požadavek na adresu /, spustí se následující funkce.
  • def read_root(): = Python funkce obsluhující tuto adresu.
  • return {"message": "Ahoj ze serveru"} = server vrátí data ve formátu JSON.
Co si z tohoto bloku odnést
  • Backend je stále Python kód.
  • Jednotlivé adresy na serveru odpovídají Python funkcím.
  • V této chvíli se ještě nečte databáze, vrací se jen pevně napsaná odpověď.
Krok 9: jak číst @app.get("/") úplně od základů

Tento řádek je pro první backend zásadní. Na první pohled vypadá zvláštně, ale ve skutečnosti spojuje jen tři jednoduché věci: objekt aplikace, HTTP metodu a cestu.

samotný dekorátor 
@app.get("/")
Rozpad po částech
  • app = objekt aplikace vytvořený přes FastAPI().
  • .get = metoda objektu app, která slouží pro HTTP požadavky typu GET.
  • "/" = cesta neboli adresa, konkrétně kořen webu.
  • @ = dekorátor, který říká, že následující funkce patří k této cestě a této metodě.
Jak to číst lidsky:
„Když někdo pošle GET požadavek na adresu /, použij funkci pod tímto řádkem.“
Co tento zápis neznamená
  • Neznamená to „get vrať lomítko“.
  • Neznamená to, že by "/" byla návratová hodnota.
  • Neznamená to, že se funkce spustí hned při načtení souboru.

FastAPI si jen zapamatuje:
pro GET na / má zavolat tuto funkci.

Krok 10: proč nestačí jen spustit python.exe

Tady vzniká velmi časté nedorozumění. Samotný python.exe je jen interpreter. To znamená program, který umí vykonávat Python kód. Sám ale neví, co konkrétně má spustit.

spuštění backendu přes interpreter z .venv 
.\.venv\Scripts\python.exe -m uvicorn main:app --reload
Rozklad příkazu po částech
  • .\.venv\Scripts\python.exe = použij Python z tohoto virtuálního prostředí.
  • -m uvicorn = spusť modul uvicorn jako program.
  • main:app = v souboru main.py najdi proměnnou app.
  • --reload = při změně souboru server restartuj.
Jak to číst lidsky:
„Použij Python z virtuálního prostředí, spusť server Uvicorn a rozběhni na něm aplikaci app ze souboru main.py.“
Co se stane, když jen klikneš na python.exe
  • Spustí se jen samotný interpreter.
  • Nedostane informaci o modulu uvicorn.
  • Nedostane informaci o souboru main.py.
  • Nedostane informaci o objektu app.

Proto se backend sám od sebe nerozběhne.

Frontend × Backend × Databáze

Většina webových aplikací se neskládá z jediné části. Aby bylo jasné, kdo co dělá, je dobré tyto části od sebe oddělit.

  • Frontend = viditelná část aplikace v prohlížeči. Zde jsou tlačítka, formuláře, seznamy a vzhled stránky.
  • Backend = logická část aplikace. Přijímá požadavky, pracuje s daty, rozhoduje, co se má stát, a vrací výsledek.
  • Databáze = uložiště dat. V tomto projektu je to soubor reservations.db.

Frontend tedy nic „neví“ o tom, jak jsou data uložená uvnitř databáze. O to se stará backend. Databáze sama nic nezobrazuje. Jen uchovává informace.

V našem projektu:
frontend rezervace zobrazí, backend je načte nebo uloží, databáze je uchová.

Co je server?

Server je program, který běží a čeká na požadavky. Jakmile nějaký požadavek přijde, server vrátí odpověď.

  • Někdo otevře webovou adresu v prohlížeči.
  • Požadavek dorazí na server.
  • Backend rozhodne, co má server vrátit.
  • Server pošle odpověď zpět.

V tomto projektu nebude server „nějaká vzdálená velká věc na internetu“. Zatím poběží lokálně na vlastním počítači. I tak je to server — jen místní, vývojový.

Důležité:
Backend je logika. Server je způsob, jak tato logika běží a zpřístupňuje se přes web.

Co je API?

API je rozhraní, přes které spolu komunikují programy. Není to tlačítko ani obrazovka. Je to domluvený způsob, jak si jedna část systému říká druhé o data nebo o akci.

  • Frontend pošle požadavek.
  • Backend ho zpracuje.
  • Backend vrátí odpověď.

V praxi to může vypadat takto:

  1. Stránka požádá o seznam rezervací.
  2. Backend otevře databázi.
  3. Načte rezervace.
  4. Vrátí data zpět.
  5. Stránka je zobrazí.
Zapamatování:
API je „komunikační okénko“ mezi frontendem a backendem.

Proč stránka neotevře SQLite databázi sama?

SQLite databáze je soubor na disku. Je to interní část aplikace, ne něco, co by prohlížeč běžně otevíral a obsluhoval.

  • Webová stránka v prohlížeči neumí používat Python modul sqlite3.
  • Frontend běží jinak než Python skript a nemá přímý přístup k backendovým knihovnám.
  • Práci s databází proto přebírá backend.
  • Backend data vrátí stránce bezpečně a řízeně.

Díky tomu zůstává projekt přehledný: stránka zobrazuje, backend zpracovává, databáze ukládá.

Zjednodušeně:
stránka si o data řekne, backend je vyzvedne z databáze, stránka je pak jen zobrazí.

GET a POST – dvě základní HTTP metody

Když spolu komunikuje prohlížeč a server, používají tzv. HTTP metody. Pro první backend jsou nejdůležitější dvě:

  • GET = požadavek na získání dat.
  • POST = požadavek na odeslání nebo vytvoření dat.

V rezervačním systému bude výpis rezervací typicky GET. Kdyby se později přidával formulář na vytvoření rezervace přes web, šlo by typicky o POST.

Proto se ve FastAPI setkáš se zápisy jako: 

@app.get("/")
@app.post("/reservations")
Zapamatování:
GET = chci něco přečíst
POST = chci něco poslat nebo vytvořit

Proč budeme používat FastAPI?

FastAPI je framework pro tvorbu backendu v Pythonu. Framework znamená hotový základ, který usnadňuje stavbu aplikace.

  • Má přehledný a krátký zápis.
  • Rychle v něm vznikne první funkční server.
  • Dobře se na něm vysvětluje vazba mezi URL adresou a Python funkcí.
  • Umí vracet JSON, HTML i další typy odpovědí.

Pro tuto výuku je klíčové, že i velmi malý program už ukáže, jak backend funguje: 

app = FastAPI()

@app.get("/")
def read_root():
    return {"message": "Ahoj ze serveru"}
Nejdůležitější myšlenka:
každá adresa na serveru může odpovídat určité Python funkci.

Proč FastAPI musíme instalovat?

Python obsahuje velké množství vestavěných modulů, ale neobsahuje všechno. FastAPI patří mezi externí balíčky.

  • Základní Python obsahuje například sqlite3.
  • FastAPI je potřeba doinstalovat.
  • Dokud není nainstalované, import from fastapi import FastAPI nebude fungovat.

Instalace probíhá přes správce balíčků pip. Nejprve se balíček nainstaluje, teprve potom ho lze používat v kódu. 

py -m pip install "fastapi[standard]"
Rozdíl:
pip install ... = něco do prostředí přidám
import ... = něco už přidaného použiji v programu

Co je virtuální prostředí?

Virtuální prostředí je oddělené prostředí pro jeden konkrétní Python projekt. Nejčastěji se vytváří jako složka .venv uvnitř projektu.

  • Má vlastní balíčky.
  • Odděluje projekt od ostatních projektů i od globální instalace Pythonu.
  • Po aktivaci používají příkazy python, pip a další právě toto prostředí.

To je důležité hlavně tehdy, když jeden projekt používá určitou verzi balíčku a jiný projekt jinou. Bez oddělení by se tyto závislosti snadno pletly.

Virtuální prostředí neobsahuje samotný projektový kód. Obsahuje hlavně technické zázemí projektu — interpreter, skripty a nainstalované balíčky.

Prakticky:
složka .venv drží závislosti pohromadě a zajišťuje, že backend běží nad správnou sadou balíčků.

Co je pip?

pip je správce Python balíčků. Umí balíčky instalovat, aktualizovat i odstraňovat.

  • install = instalace balíčku
  • show = zobrazení informací o balíčku
  • uninstall = odinstalace

V praxi je výhodné používat zápis: 

py -m pip install "fastapi[standard]"

Tím je jasné, že se pip spouští přes konkrétní Python interpreter, který je právě použitý. Je to bezpečnější než spoléhat jen na samostatný příkaz pip.

Zapamatování:
pip = nástroj na správu balíčků

Balíček × modul

Tyto pojmy se často pletou, ale je dobré je rozlišit:

  • Balíček = něco, co se instaluje přes pip.
  • Modul = něco, co se v Pythonu načítá pomocí import.
py -m pip install "fastapi[standard]"
from fastapi import FastAPI

První řádek instaluje. Druhý řádek používá to, co už bylo nainstalované.

Příklad:
FastAPI je balíček, který se instaluje. Uvnitř něj je modul fastapi, který se importuje.

py × python ve Windows

Ve Windows mohou fungovat oba příkazy, ale nemusí to být vždy stejně spolehlivé.

  • python = běžný příkaz pro spuštění Pythonu.
  • py = Windows launcher / manager pro Python.

V některých instalacích funguje bez problémů python. V jiných může být spolehlivější py. Pokud python v PowerShellu nefunguje, ale py ano, je rozumné používat ve Windows právě py. 

py -m venv .venv
py -m pip install "fastapi[standard]"
Praktické pravidlo:
ve Windows příkladech může být bezpečnější držet se py.

Aktivace ve Windows PowerShellu

Když se vytvoří virtuální prostředí ve složce .venv, Windows do něj uloží skripty do podsložky Scripts.

Pro PowerShell je správná aktivace: 

.\.venv\Scripts\Activate.ps1

Tento příkaz změní aktuální shell tak, aby používal Python a nástroje z právě vytvořeného prostředí.

  • Po aktivaci se obvykle v promptu objeví něco jako (.venv).
  • Příkaz python nebo py pak typicky pracuje v kontextu tohoto prostředí.
  • Stejně tak i balíčky instalované přes pip.

Důležité je nepoužívat unixový zápis source .venv/bin/activate, protože ten do PowerShellu nepatří.

Co dělat, když PowerShell blokuje Activate.ps1

Ve Windows může PowerShell blokovat spouštění některých skriptů. Pak se může stát, že aktivace prostředí skončí chybou.

Jedno z běžných řešení je: 

Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser

Tím se povolí spouštění lokálních skriptů pro aktuálního uživatele. Potom se dá zkusit aktivace znovu: 

.\.venv\Scripts\Activate.ps1
Když nechceš řešit aktivaci:
stále lze prostředí použít i bez aktivace přímým spuštěním .\.venv\Scripts\python.exe.

Proč aktivace není vždy nutná

Aktivace prostředí je pohodlná, ale není technicky povinná.

Důvod je jednoduchý: ve složce virtuálního prostředí už existuje vlastní Python interpreter a vlastní spouštěcí soubory balíčků. 

.\.venv\Scripts\python.exe -m pip install "fastapi[standard]"
.\.venv\Scripts\python.exe -m uvicorn main:app --reload

Když se používá tato úplná cesta, je jedno, jestli je prostředí aktivované. Příkazy míří přímo do správného prostředí.

  • Je to užitečné při potížích s PowerShellem.
  • Je to užitečné, když nefunguje PATH.
  • Je to užitečné, když je potřeba mít jistotu, že se používá právě toto prostředí.

Co je route / endpoint?

Route nebo endpoint je konkrétní adresa na serveru, kterou backend obsluhuje.

  • / = domovská adresa
  • /reservations = například adresa pro výpis rezervací

Zápis @app.get("/") znamená: „Když někdo otevře adresu /, spusť tuto funkci.“

Je důležité vidět, že zde jsou spojené dvě věci:

  • adresa na webu
  • konkrétní Python funkce

Tím se backend mapuje na webové adresy.

Co je JSON?

JSON je běžný textový formát pro předávání dat mezi frontendem a backendem. 

{
  "message": "Ahoj ze serveru"
}

Vypadá podobně jako Python slovník, ale na webu se používá jako standardní datový formát.

  • klíč = například "message"
  • hodnota = například "Ahoj ze serveru"

Když FastAPI vrátí Python slovník, často ho automaticky převede právě na JSON odpověď.

Zapamatování:
JSON je běžný „jazyk dat“ pro webovou komunikaci.

Jak číst @app.get("/") opravdu podrobně

Tento zápis je dekorátor. To znamená, že upravuje chování funkce, která je napsaná hned pod ním. 

@app.get("/")
def read_root():
    return {"message": "Ahoj ze serveru"}
Rozpad po částech
  • app = objekt aplikace vytvořený přes FastAPI().
  • .get = metoda objektu app, která souvisí s HTTP metodou GET.
  • "/" = cesta, tedy hlavní adresa webu.
  • @ = říká, že následující funkce patří k tomuto nastavení.
Co to znamená lidsky

„Když někdo pošle GET požadavek na adresu /, použij funkci read_root().“

Co je důležité neplést
  • Není to „get vrať lomítko“.
  • "/" není návratová hodnota.
  • Funkce se při načtení souboru hned nespouští.

FastAPI si jen zapamatuje mapování: 

GET + /  →  read_root()
Praktické čtení do hodin:
„Pro adresu / a metodu GET použij funkci pod tímto řádkem.“

Proč nestačí spustit jen python.exe?

python.exe je interpreter. To znamená program, který umí vykonávat Python kód. Sám o sobě ale neví, co konkrétně má spustit.

Když jen spustíš python.exe
  • spustí se Python
  • ale nedostane informaci o main.py
  • nedostane informaci o uvicorn
  • nedostane informaci o objektu app

Proto se backend sám od sebe nerozběhne.

Když spustíš
.\.venv\Scripts\python.exe -m uvicorn main:app --reload

pak už Pythonu říkáš přesně:

  • jakého Pythonu má použít
  • jaký modul má spustit
  • jaký soubor má načíst
  • jaký objekt aplikace má hledat
Přirovnání:
samotný python.exe je jako zapnutí motoru. Až zbytek příkazu říká, kam se má jet.

Jak číst příkaz python -m uvicorn main:app --reload

.\.venv\Scripts\python.exe -m uvicorn main:app --reload
Rozpad po částech
  • .\.venv\Scripts\python.exe = použij Python z tohoto virtuálního prostředí.
  • -m = spusť určitý modul jako program.
  • uvicorn = modul serveru, který FastAPI aplikaci rozběhne.
  • main:app = v souboru main.py najdi proměnnou app.
  • --reload = při změně souboru server automaticky restartuj.
Jak to číst lidsky

„Použij Python z virtuálního prostředí, spusť Uvicorn a na něm rozběhni aplikaci app ze souboru main.py.“

Klíčová myšlenka:
Python sám o sobě backend nespouští. Backend se spustí až tehdy, když Python dostane přesný pokyn, co má načíst a co nad tím má rozběhnout.

Co je Uvicorn?

Uvicorn je program, který umí spustit Python webovou aplikaci jako server.

  • Bez něj by soubor main.py nebyl webovým serverem.
  • Uvicorn vezme objekt app a zpřístupní ho přes webovou adresu.
  • FastAPI CLI ho používá uvnitř pro vývojové spuštění.

Proto můžeš backend spouštět buď vyšší vrstvou: 

.\.venv\Scripts\fastapi.exe dev main.py

nebo přímo přes Uvicorn: 

.\.venv\Scripts\python.exe -m uvicorn main:app --reload

Co znamená 127.0.0.1 nebo localhost?

127.0.0.1 je adresa vlastního počítače. Název localhost je lidsky čitelný ekvivalent.

  • http://127.0.0.1:8000 = webový server běžící na tomto počítači
  • 8000 = port, tedy konkrétní komunikační „brána“ serveru

Když backend běží lokálně, neznamená to, že je automaticky veřejně dostupný na internetu. Zatím je dostupný jen na tomto počítači.

Proč oba programy vidí stejná data?

Protože konzolový program i backend pracují se stejným souborem reservations.db.

  • Konzole data zapisuje.
  • Backend stejná data čte.
  • Jedna databáze tedy slouží více částem téhož projektu.

To je velmi důležitá vlastnost celého projektu. Není potřeba mít dvě databáze ani dvě různé sady dat. Jediný zdroj pravdy je jeden společný SQLite soubor.

Důsledek:
co se uloží v konzoli, může být o chvíli později vidět na stránce.
