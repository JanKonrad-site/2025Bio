6.1.5 Backend – cesty serveru, HTML, JSON a 404

V minulé kapitole jsme rozjeli úplný základ serveru a první textovou odpověď. Teď server naučíme rozlišovat, o jakou adresu si klient říká, a podle toho vrátit buď HTML stránku, JSON data z databáze, nebo chybovou odpověď 404.

Co se naučíš?

  • Rozpoznání cesty požadavku – pochopíš, jak server pozná, jestli klient chce stránku, API data nebo neexistující adresu.
  • Metody _send_json() a _send_file() – zjistíš, jak server pošle JSON nebo celý HTML soubor.
  • Skutečné do_GET(self) – uvidíš, jak se skládá hlavní rozhodovací logika serveru.
  • Propojení s frontendem – pochopíš, proč frontend volá /api/movies a proč server posílá index.html.
Tip: Tohle je první chvíle, kdy backend opravdu „mluví“ s frontendem. Jedna cesta vrací stránku, druhá data a třetí vrací chybu.

1) Proč server musí rozlišovat různé cesty

Když uživatel otevře server v prohlížeči, neznamená to vždy totéž. Jednou chce načíst samotnou stránku, jindy chce JavaScript na stránce načíst data z API. A někdy uživatel nebo prohlížeč požádá o cestu, která v našem projektu vůbec neexistuje.

  1. Když klient otevře /, chceme vrátit HTML stránku.
  2. Když frontend zavolá /api/movies, chceme vrátit data jako JSON.
  3. Když přijde jiná cesta, chceme vrátit chybovou odpověď 404.
  • Jeden server tedy může obsluhovat více „adres“.
  • Každá adresa může vracet jiný typ obsahu.
  • Právě to je základ routování na serveru.
2) První blok kódu – metoda _send_json()

Ve finálním backendu je připravená pomocná metoda pro JSON odpověď. Ta se používá hlavně pro API cestu /api/movies. :contentReference[oaicite:2]{index=2} 

def _send_json(self, data, status=200):
    body = json.dumps(data, ensure_ascii=False).encode("utf-8")

    self.send_response(status)
    self.send_header("Content-Type", "application/json; charset=utf-8")
    self.send_header("Content-Length", str(len(body)))
    self.end_headers()
    self.wfile.write(body)
  1. data jsou Python data, která chceme poslat klientovi.
  2. json.dumps(...) je převede na JSON text.
  3. ensure_ascii=False dovolí zachovat české znaky bez zbytečného přepisování.
  4. encode("utf-8") převede text na bajty.
  5. Content-Type říká klientovi, že odpověď je JSON.
  6. wfile.write(body) pošle hotové tělo odpovědi.
  • Tato metoda je podobná _send_text(), ale posílá JSON.
  • Klient pak ví, že má odpověď číst jako data, ne jako obyčejný text.
  • Právě to frontend očekává při volání fetch("/api/movies"). :contentReference[oaicite:3]{index=3}
3) Druhý blok kódu – metoda _send_file()

Když klient chce samotnou stránku, server nevrací JSON, ale celý HTML soubor. K tomu slouží tato metoda. :contentReference[oaicite:4]{index=4} 

def _send_file(self, filepath: Path, status=200, content_type="text/html; charset=utf-8"):
    if not filepath.exists():
        self._send_text("Soubor nenalezen.", status=404)
        return

    body = filepath.read_bytes()
    self.send_response(status)
    self.send_header("Content-Type", content_type)
    self.send_header("Content-Length", str(len(body)))
    self.end_headers()
    self.wfile.write(body)
  1. filepath je cesta k souboru, který chceme poslat.
  2. if not filepath.exists(): ověří, jestli soubor opravdu existuje.
  3. Když soubor chybí, vrátí se textová chyba se stavem 404.
  4. read_bytes() načte celý soubor jako bajty.
  5. Content-Type říká klientovi, že jde o HTML obsah.
  6. wfile.write(body) pošle celý soubor klientovi.
  • Tady server nečte databázi.
  • Tady server jen vezme soubor z disku a pošle ho do prohlížeče.
  • To je přesně to, co potřebujeme pro index.html.
4) Třetí blok kódu – rozpoznání cesty přes urlparse(self.path)

Aby server věděl, co vrátit, musí se nejdřív podívat, jakou cestu klient požaduje. Ve finálním backendu to začíná takto: :contentReference[oaicite:5]{index=5} 

parsed = urlparse(self.path)
path = parsed.path
  1. self.path obsahuje cestu z příchozího HTTP požadavku.
  2. urlparse(...) tuto cestu rozebere na části.
  3. parsed.path vytáhne jen čistou cestu bez dalších částí.
  4. Výsledek uložíme do proměnné path, se kterou se pak snadno pracuje.
  • Když klient požádá o /api/movies, proměnná path bude mít tuto hodnotu.
  • Když klient otevře hlavní stránku, cesta bude třeba /.
  • Podle této hodnoty potom server rozhoduje.
5) Čtvrtý blok kódu – skutečné do_GET(self)

Tohle je hlavní rozhodovací část serveru. Přesně takto je logika rozepsaná i ve finálním backendu. :contentReference[oaicite:6]{index=6} 

def do_GET(self):
    parsed = urlparse(self.path)
    path = parsed.path

    # 1) API endpoint: /api/movies
    if path == "/api/movies":
        data = get_movies()
        self._send_json(data)
        return

    # 2) Frontend: / nebo /index.html
    if path == "/" or path == "/index.html":
        self._send_file(Path(__file__).resolve().with_name("index.html"))
        return

    # 3) všechno ostatní: 404
    self._send_text("404 Not Found", status=404)

Tohle je velmi důležitý moment. Poprvé je tu na jednom místě vidět propojení backendu, databáze i frontendu.

5.1 Cesta /api/movies

Když klient chce /api/movies, backend zavolá get_movies() a pošle výsledek jako JSON přes _send_json(...). :contentReference[oaicite:7]{index=7}

  • Tady se propojí databáze s API.
  • Data jdou z Pythonu do JSONu.
  • To je přesně cesta, kterou používá frontend. :contentReference[oaicite:8]{index=8}
5.2 Cesta / nebo /index.html

Když klient otevře hlavní adresu serveru, backend pošle soubor index.html.

  • Tady server nevrací JSON.
  • Tady vrací skutečnou webovou stránku.
  • Právě díky tomu se stránka zobrazí v prohlížeči.
5.3 Všechno ostatní = 404

Když cesta neodpovídá žádné známé variantě, server vrátí text 404 Not Found a stav 404.

  • To znamená „nenalezeno“.
  • Klient tak pozná, že daná cesta na serveru neexistuje.
  • To je normální a správné chování webového serveru.
6) Jak se to propojí s frontendem

Ve frontendu se při načtení stránky volá tento kód: :contentReference[oaicite:9]{index=9} 

const res = await fetch("/api/movies");
const movies = await res.json();
  1. Uživatel otevře server v prohlížeči.
  2. Server vrátí index.html.
  3. Uvnitř stránky se spustí JavaScript.
  4. JavaScript zavolá fetch("/api/movies").
  5. Server rozpozná cestu /api/movies.
  6. Backend zavolá get_movies() a pošle JSON.
  7. Frontend tento JSON přečte a vypíše do tabulky.
  • Tady už je krásně vidět celý směr toku dat.
  • Frontend si data nevymýšlí sám.
  • Vždy si je vyžádá od backendu.
7) Co máš mít po této kapitole hotové

Po této kapitole už server umí rozlišit hlavní typy požadavků, které náš jednoduchý informační systém potřebuje.

  1. Máš metodu _send_json() pro JSON odpověď.
  2. Máš metodu _send_file() pro poslání HTML souboru.
  3. Máš v do_GET() logiku pro /api/movies.
  4. Máš v do_GET() logiku pro / a /index.html.
  5. Máš i výchozí chování pro neexistující cesty: 404.
  • Tohle už je skutečný základ webového backendu.
  • V další lekci se všechno finálně spojí a spustí jako celek.
  • Tam už půjdeme od startu serveru až po vykreslenou tabulku v prohlížeči.

Shrnutí

  • Žák chápe, že server rozlišuje různé cesty podle adresy požadavku.
  • Žák ví, že _send_json() posílá data pro API a _send_file() posílá HTML soubor.
  • Žák rozumí tomu, že /api/movies vrací JSON a / nebo /index.html vrací stránku.
  • Žák ví, proč existuje odpověď 404 Not Found.

Proč server potřebuje více cest

Jedna webová aplikace téměř nikdy nevrací jen jednu jedinou odpověď na všechno.

  • hlavní stránka vrací HTML,
  • API vrací data,
  • neplatná cesta vrací chybu.

Rozlišování cest je proto základní povinnost serveru.

Rozbor _send_json() po krocích

Tato metoda vezme Python data a pošle je klientovi jako JSON odpověď.

  • json.dumps(...) převádí Python data na JSON text,
  • Content-Type říká, že jde o JSON,
  • wfile.write(...) pošle tělo odpovědi klientovi.

Rozbor _send_file() po krocích

Tato metoda vezme soubor z disku a pošle ho klientovi.

  • nejdřív ověří, jestli soubor existuje,
  • pak načte jeho obsah jako bajty,
  • nakonec nastaví hlavičky a soubor odešle.

Je vhodná hlavně pro HTML soubory a další statický obsah.

Co přesně dělá urlparse(self.path)

self.path obsahuje cestu z příchozího požadavku. Funkce urlparse(...) ji rozloží na části a my si pak vezmeme čistou cestu přes parsed.path.

  • To je přehlednější než pracovat se syrovým řetězcem.
  • Server pak snadno porovnává, jestli jde o / nebo třeba /api/movies.

Jak číst do_GET() jako celek

Celou logiku můžeš číst takto:

  • zjisti, jakou cestu klient chce,
  • když chce API, vrať JSON,
  • když chce stránku, vrať HTML soubor,
  • jinak vrať 404.

To je hlavní routovací logika našeho jednoduchého serveru.

Minihra – sestav routování serveru

Seřaď kroky tak, aby server správně rozlišil, kdy má vrátit JSON data, kdy HTML stránku a kdy chybovou odpověď 404.

Sestav metodu do_GET()

Kliknutím označíš kartičku, druhým kliknutím ji prohodíš s jinou. Když bude pořadí správně, ukáže se výsledný Python kód.

Výsledný kód – routování serveru 
def do_GET(self):
    parsed = urlparse(self.path)
    path = parsed.path

    if path == "/api/movies":
        data = get_movies()
        self._send_json(data)
        return

    if path == "/" or path == "/index.html":
        self._send_file(Path(__file__).resolve().with_name("index.html"))
        return

    self._send_text("404 Not Found", status=404)
