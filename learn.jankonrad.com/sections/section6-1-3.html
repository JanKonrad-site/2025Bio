6.1.2 Backend bez frameworku – co už Python umí sám

V minulé části vznikla databáze reservations.db a konzolový program, který do ní umí zapisovat rezervace. Než do projektu přidáme framework, je metodicky lepší pochopit, že velmi jednoduchý backend jde postavit i pomocí toho, co už Python obsahuje v základu.

V této lekci si proto ukážeme, že: SQLite ukládá data, jednoduchý HTTP server přijímá požadavky a Python kód mezi tím tvoří backendovou logiku. Až v další lekci na to navážeme FastAPI, které celý zápis zjednoduší.

Co se v této lekci naučíš

  • Že backend není automaticky totéž co framework.
  • Že Python už v základu obsahuje modul sqlite3 pro práci s databází.
  • Že Python už v základu obsahuje i jednoduchý modul http.server pro webový server.
  • Jak spolu souvisí databáze, server, backend a frontend.
  • Jak vypadá nejmenší možný backend bez FastAPI.
  • Jak backend vrátí odpověď ve formátu JSON.
  • Jak backend přečte data z reservations.db a pošle je stránce.
  • Proč i přesto budeme v další lekci používat FastAPI.
Cíl kapitoly:
Pochopit backend nejdřív bez „kouzla frameworku“. Student má nejprve vidět čistý princip: požadavek → Python funkce → databáze → odpověď.

Na co tato kapitola navazuje

  1. Existuje databáze reservations.db.
  2. Existuje konzolový Python program, který do ní zapisuje rezervace.
  3. Další krok je zpřístupnit tato data přes web.
  4. Nejdřív si ukážeme princip bez frameworku.
  5. Teprve potom přejdeme na pohodlnější řešení ve FastAPI.
Jedna věta, kterou si zapamatuj:
Backend není název konkrétní knihovny. Backend je role programu v projektu.
Krok 1: co je backend a proč to nemusí být hned FastAPI

Když se řekne backend, mnoho začátečníků si hned představí nějaký framework. To ale není přesné. Backend je především část aplikace, která zpracovává logiku a pracuje s daty.

  • Frontend = to, co vidí uživatel v prohlížeči.
  • Backend = Python program, který rozhoduje, co se má stát.
  • Databáze = místo, kde jsou uložená data.
  • Server = běžící program, který čeká na požadavky a vrací odpovědi.
Důležité:
Backend můžeš napsat i bez externího frameworku. Framework je až další vrstva, která práci usnadňuje.
Krok 2: co už Python umí sám – sqlite3 a http.server

Python už obsahuje velkou standardní knihovnu. V našem projektu jsou teď důležité hlavně dva moduly:

  • sqlite3 = práce s SQLite databází.
  • http.server = jednoduchý HTTP server.
Pozor na velmi důležitý rozdíl:
sqlite3 není náhrada za webový server. sqlite3 řeší ukládání a čtení dat. http.server řeší příjem webových požadavků.
Co dělá každý z těchto modulů
  • sqlite3 otevře databázový soubor a pracuje s tabulkami a daty.
  • http.server naslouchá na adrese a portu, například 127.0.0.1:8000.
  • Backend propojí obě věci dohromady.
  • Frontend si pak říká backendu o data přes HTTP.
Krok 3: nejmenší možný backend bez frameworku

Nejdřív si ukažme úplné minimum: server, který běží na počítači a na adrese / vrátí jednoduchou JSON odpověď.

server_basic.py – úplné minimum 
from http.server import BaseHTTPRequestHandler, HTTPServer
import json


class MyHandler(BaseHTTPRequestHandler):
    def do_GET(self):
        if self.path == "/":
            data = {"message": "Ahoj ze serveru"}
            body = json.dumps(data, ensure_ascii=False).encode("utf-8")

            self.send_response(200)
            self.send_header("Content-Type", "application/json; charset=utf-8")
            self.send_header("Content-Length", str(len(body)))
            self.end_headers()
            self.wfile.write(body)
        else:
            self.send_response(404)
            self.end_headers()


server = HTTPServer(("127.0.0.1", 8000), MyHandler)
print("Server běží na adrese: http://127.0.0.1:8000")
server.serve_forever()
Vysvětlení kódu „slovo od slova“
  • from http.server import BaseHTTPRequestHandler, HTTPServer = importujeme třídy pro jednoduchý HTTP server.
  • import json = budeme vracet data ve formátu JSON.
  • class MyHandler(...) = definujeme, jak se server zachová při požadavku.
  • def do_GET(self): = co se má stát při HTTP metodě GET.
  • if self.path == "/" = reagujeme na domovskou adresu.
  • json.dumps(...) = převedeme Python slovník na JSON text.
  • self.send_response(200) = vrátíme stav OK.
  • HTTPServer(("127.0.0.1", 8000), MyHandler) = spustíme server na konkrétní adrese a portu.
  • serve_forever() = server běží a čeká na další požadavky.
To nejdůležitější:
Už tady existuje backend. Není zde FastAPI, ale je zde Python server, který umí přijmout požadavek a vrátit odpověď.

Krok 5: backend, který otevře databázi a vrátí rezervace

Teď přidáme to nejdůležitější: server nebude vracet jen pevně zapsanou zprávu, ale skutečná data z databáze reservations.db.

server_reservations.py – čtení SQLite databáze 
from http.server import BaseHTTPRequestHandler, HTTPServer
from pathlib import Path
import sqlite3
import json


DB_FILE = Path(__file__).with_name("reservations.db")


class ReservationHandler(BaseHTTPRequestHandler):
    def do_GET(self):
        if self.path == "/reservations":
            conn = sqlite3.connect(DB_FILE)
            conn.row_factory = sqlite3.Row

            rows = conn.execute("""
                SELECT id, name, date, time
                FROM reservations
                ORDER BY id DESC
            """).fetchall()

            conn.close()

            data = [dict(row) for row in rows]
            body = json.dumps(data, ensure_ascii=False).encode("utf-8")

            self.send_response(200)
            self.send_header("Content-Type", "application/json; charset=utf-8")
            self.send_header("Content-Length", str(len(body)))
            self.end_headers()
            self.wfile.write(body)

        elif self.path == "/":
            data = {"message": "Server běží", "endpoint": "/reservations"}
            body = json.dumps(data, ensure_ascii=False).encode("utf-8")

            self.send_response(200)
            self.send_header("Content-Type", "application/json; charset=utf-8")
            self.send_header("Content-Length", str(len(body)))
            self.end_headers()
            self.wfile.write(body)

        else:
            self.send_response(404)
            self.end_headers()


server = HTTPServer(("127.0.0.1", 8000), ReservationHandler)
print("Server běží na adrese: http://127.0.0.1:8000")
server.serve_forever()
Co se v tomto kódu děje
  1. Server přijme požadavek na adresu /reservations.
  2. Python otevře soubor reservations.db.
  3. Spustí SQL dotaz nad tabulkou reservations.
  4. Výsledek převede na seznam slovníků.
  5. Seznam odešle jako JSON odpověď.
Tady už je vidět celý princip backendu:
webový požadavek spustí Python kód, ten otevře databázi, načte data a vrátí je přes web zpět.
Krok 6: proč frontend nepůjde přímo do SQLite databáze

Tohle je velmi důležitý metodický bod. Student by mohl dostat nápad: „Když máme databázi jako soubor, proč ji prostě neotevřít rovnou ze stránky?“

  • Protože prohlížeč neběží jako Python skript.
  • Protože frontend neumí použít Python modul sqlite3.
  • Protože databáze je interní vrstva projektu, ne něco, co má přímo obsluhovat prohlížeč.
  • Proto je mezi stránkou a databází potřeba backend.
Zjednodušeně:
stránka si o data řekne, backend je načte, stránka je jen zobrazí.
Krok 7: kde se v tom všem objeví virtuální prostředí

V této lekci ještě virtuální prostředí technicky nepotřebujeme, protože jsme zatím nepřidali žádný externí balíček. Ale je dobré už teď chápat, proč se brzy objeví.

  • Když používáme jen standardní knihovnu, žádný externí balíček neinstalujeme.
  • Jakmile ale přidáme FastAPI, už budeme chtít oddělené prostředí projektu.
  • Virtuální prostředí drží projektové závislosti pohromadě.
  • Tím zabrání tomu, aby se různé projekty navzájem pletly.
Windows PowerShell – vytvoření prostředí pro další lekci 
py -m venv .venv
Teď ještě ne:
V této lekci jde hlavně o pochopení principu. Virtuální prostředí bude prakticky důležité až ve chvíli, kdy začneme instalovat FastAPI.
Krok 8: proč i přesto přejdeme v další lekci na FastAPI

Jednoduchý backend bez frameworku je výborný pro pochopení principu. Ale jakmile začne projekt růst, ruční práce rychle přibývá.

  • Musíme ručně zpracovávat cesty jako / nebo /reservations.
  • Musíme ručně skládat odpovědi a hlavičky.
  • Musíme si sami držet pořádek v obsluze požadavků.
  • Kód je delší a méně pohodlný na rozšiřování.
Proto bude další krok tento:
stejný princip zachováme, ale použijeme FastAPI, které zápis zkrátí a zpřehlední.
Co bude ve FastAPI pohodlnější
  • kratší zápis rout a endpointů,
  • čistší práce s JSON odpověďmi,
  • lepší struktura backendu,
  • automatická dokumentace API.

Co už po této lekci víš

  • Backend není automaticky framework, ale role programu v aplikaci.
  • Python už v základu obsahuje sqlite3 i http.server.
  • sqlite3 řeší databázi, ale ne webové API.
  • http.server umí přijmout HTTP požadavek a vrátit odpověď.
  • Backend může zpracovat požadavek, otevřít databázi a vrátit JSON.
  • Frontend nepracuje přímo s databází, ale přes backend.
  • Jednoduchý backend bez frameworku je výborný pro pochopení principu.
  • V další lekci použijeme FastAPI jako pohodlnější a čistší zápis stejné myšlenky.

Backend není framework

Backend je část aplikace, která zpracovává logiku a pracuje s daty. Framework je až nástroj, který tuto práci usnadňuje.

  • Backend můžeš napsat i čistě v Pythonu.
  • Framework není definice backendu.
  • Framework je pomocník pro pohodlnější zápis.
Zapamatování:
Backend = role programu
Framework = nástroj, který backend pomáhá psát

Frontend × Backend × Databáze

  • Frontend = stránka v prohlížeči.
  • Backend = Python logika aplikace.
  • Databáze = uložená data projektu.

Frontend zobrazuje, backend rozhoduje a databáze ukládá.

V našem projektu:
frontend ukáže rezervace, backend je načte, databáze je uchová.

Co je sqlite3?

sqlite3 je Python modul pro práci se SQLite databází. Umožňuje otevřít databázový soubor, spouštět SQL dotazy a číst nebo ukládat data.

  • Otevírá databázi.
  • Spouští SQL příkazy.
  • Vrací řádky z tabulek.
  • Ukládá změny přes commit().

Co je http.server?

http.server je jednoduchý modul standardní knihovny, který umí vytvořit základní HTTP server.

  • Naslouchá na adrese a portu.
  • Přijímá požadavky typu GET.
  • Umožní vrátit odpověď klientovi.
Pozor:
Pro výuku je skvělý, ale pro větší nebo produkční aplikaci není ideální.

Co je JSON?

JSON je textový formát pro přenos dat mezi klientem a serverem. 

{
  "message": "Ahoj ze serveru"
}

Backend často vrací data právě takto, protože je frontend umí snadno zpracovat.

Co je route / adresa serveru?

Route je konkrétní cesta na serveru, například / nebo /reservations.

  • / = domovská adresa serveru
  • /reservations = výpis rezervací

Server podle této adresy pozná, jakou část kódu má spustit.

Proč stránka neotevře SQLite databázi sama?

Protože prohlížeč neběží jako Python prostředí a neumí používat backendový modul sqlite3.

  • Frontend zobrazuje.
  • Backend zpracovává.
  • Databáze ukládá.

Tato dělba rolí drží projekt přehledný a bezpečnější.

Proč oba programy vidí stejná data?

Protože konzolový program i server pracují se stejným souborem reservations.db.

  • Konzole data zapisuje.
  • Server stejná data čte.
  • Jedna databáze je společný zdroj pravdy.

Proč venv přijde až s FastAPI?

Dokud používáme jen standardní knihovnu, nic dalšího neinstalujeme.

Jakmile ale přidáme FastAPI a další externí balíčky, bude se hodit oddělit prostředí projektu do .venv.

Smysl:
nejdřív pochopit princip, potom zavést projektové prostředí a framework.

Proč po této lekci přejdeme na FastAPI?

Protože princip už bude jasný. A v tu chvíli už není důvod psát všechno ručně.

  • FastAPI zkrátí kód.
  • Zpřehlední routy a odpovědi.
  • Usnadní další rozšiřování projektu.
  • Přidá automatickou dokumentaci API.
Pořadí výuky:
nejdřív pochopení, potom pohodlnější nástroj
