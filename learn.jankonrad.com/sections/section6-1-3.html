6.1.2 Co je backend a jak funguje server

V předchozí části už vznikla data projektu a teď přichází další velké téma: backend.

Tato část není hlavně o databázi. Databáze je jen jedna součást systému. Hlavní cíl je pochopit, co je server, co je backend, jak mezi sebou mluví prohlížeč a Python, co je požadavek, co je odpověď a jak se z obyčejného Python skriptu stane program, který naslouchá na adrese a vrací data přes web.

Nejdřív se vše vysvětlí obecně a pomalu. Teprve potom se ten princip ukáže na našem souboru server_basic.py.

Co se v této kapitole opravdu objasní

  • Co je backend.
  • Co je server a proč to není totéž co backend.
  • Co znamená, že server „naslouchá“.
  • Co je požadavek a co je odpověď.
  • Co je HTTP a proč se používá právě na webu.
  • Co přesně znamená metoda GET.
  • Jak vypadá tok komunikace mezi prohlížečem, serverem a další částí aplikace.
  • Jak Python pozná, kterou část kódu má při požadavku spustit.
  • Jak se z adresy / nebo /reservations stane konkrétní odpověď.
  • Jak tento princip vypadá v souboru server_basic.py.
Nejdůležitější myšlenka celé kapitoly:
Backend je vrstva logiky. Server je způsob, jak tato logika běží a jak je dostupná přes web.
Blok 1: co je backend

Slovo backend neoznačuje konkrétní knihovnu ani konkrétní programovací jazyk. Není to automaticky FastAPI, Flask, Django ani nic jiného. Backend je hlavně role v aplikaci.

Když se řekne backend, myslí se tím ta část programu, která:

  • přijímá požadavky,
  • rozhoduje, co se má stát,
  • provádí logiku aplikace,
  • pracuje s daty nebo jinými částmi systému,
  • a nakonec vrací výsledek.

Backend tedy není to, co se zobrazuje na stránce. Backend je to, co stojí „za stránkou“.

Zjednodušeně:
Frontend ukazuje. Backend rozhoduje.

Příklad bez technických detailů

Představ si rezervační stránku. Na obrazovce je tlačítko „Zobrazit rezervace“. Samotné tlačítko je frontend. Ale to, co po stisknutí rozhodne, odkud se mají data načíst a co se má vrátit, to už je backend.

Blok 2: co je server a proč to není totéž co backend

Tady je důležité nepomíchat dva pojmy: backend a server.

Backend je logika. Server je program, který běží, čeká na požadavky a předává je backendové logice.

  • Backend = co se má stát.
  • Server = program, který požadavek přijme a umožní odpověď odeslat zpět.

Jinak řečeno: backend je „mozek“ rozhodování, server je „komunikační brána“, přes kterou je backend dostupný.

Praktická představa

Když otevřeš v prohlížeči nějakou adresu, neznamená to, že prohlížeč mluví přímo s nějakou funkcí v Pythonu. Mluví se serverem. Server požadavek přijme, předá ho správné části backendu a potom vrátí odpověď.

Jedna velmi přesná věta:
Server je běžící program, který naslouchá na adrese a portu a umí přijmout a vracet HTTP komunikaci.
Blok 3: co znamená, že server „běží“ a „naslouchá“

Když se řekne, že server běží na adrese http://127.0.0.1:8000, znamená to, že existuje aktivní proces, který:

  • je spuštěný,
  • čeká na síťovou komunikaci,
  • naslouchá na určitém portu,
  • a je připraven reagovat na příchozí požadavky.

Co je adresa a co je port

  • 127.0.0.1 = tento počítač, tedy localhost.
  • 8000 = port, tedy konkrétní komunikační místo.

To slovo naslouchat znamená, že server neustále čeká: „Přijde nějaký požadavek? Chce někdo nějakou adresu? Mám na něco odpovědět?“

Přesná představa

Server nesedí a „nespouští se znovu při každém kliknutí“. Server už běží a jednotlivé požadavky přicházejí k němu. On je jeden po druhém zpracovává.

Blok 4: co je požadavek a co je odpověď

Webová komunikace stojí na velmi jednoduché myšlence:

  1. Klient pošle požadavek.
  2. Server požadavek přijme.
  3. Backend rozhodne, co se má stát.
  4. Server vrátí odpověď.

V našem případě je klientem nejčastěji prohlížeč.

Co je požadavek

Požadavek je zpráva, kterou klient posílá serveru. V té zprávě je například:

  • jaká adresa se požaduje,
  • jaká HTTP metoda se používá,
  • případně další informace v hlavičkách,
  • a někdy i data v těle požadavku.

Co je odpověď

Odpověď je zpráva, kterou server vrací zpět klientovi. V odpovědi bývá:

  • stavový kód, například 200 nebo 404,
  • hlavičky, například typ vráceného obsahu,
  • a samotné tělo odpovědi, tedy data.
Úplný základ webu:
požadavek → zpracování → odpověď
Blok 5: co je HTTP a proč ho používáme

Aby si klient a server rozuměli, potřebují společný jazyk. Tím jazykem je na webu nejčastěji HTTP.

HTTP určuje pravidla, jak má vypadat požadavek a jak má vypadat odpověď.

Příklad velmi zjednodušeného požadavku

Příklad HTTP požadavku 
GET /reservations HTTP/1.1

Co znamená každý kus

  • GET = jaký druh požadavku přichází.
  • /reservations = kterou cestu klient požaduje.
  • HTTP/1.1 = podle jaké verze pravidel se komunikace vede.

Díky HTTP server ví, co se po něm chce, a klient ví, jak má číst odpověď.

Blok 6: co přesně znamená GET

GET je HTTP metoda. Její význam je: chci získat odpověď nebo data.

To je velmi důležité. GET není „název nějaké Python funkce“. Je to informace, kterou posílá klient serveru.

Jak o tom přemýšlet správně

  • GET / = chci to, co je na kořenové adrese.
  • GET /reservations = chci seznam rezervací.

Co GET obvykle dělá

  • načítá stránku,
  • načítá JSON data,
  • načítá obrázek,
  • načítá seznam položek.

Co GET typicky nedělá

  • nevytváří novou rezervaci,
  • neposílá formulářová data jako hlavní účel,
  • nemění obsah databáze jako základní význam.
Nejpřesnější zkratka:
GET = „dej mi výsledek“

Příklad z běžného života

Otevření stránky v prohlížeči je ve většině případů GET. Kliknutí na odkaz je ve většině případů GET. Načtení seznamu dat z API je velmi často GET.

Blok 7: co se stane od napsání adresy až po odpověď

Teď přijde úplně nejdůležitější tok celé kapitoly. Co se stane, když se do prohlížeče zadá třeba:

Adresa rezervací 
http://127.0.0.1:8000/reservations
  1. Prohlížeč vytvoří HTTP požadavek.
  2. Do požadavku zapíše, že jde o GET.
  3. Do požadavku zapíše cestu /reservations.
  4. Pošle tento požadavek na adresu 127.0.0.1 a port 8000.
  5. Server, který na tomto portu běží, požadavek přijme.
  6. Server zjistí, že jde o GET požadavek.
  7. Proto zavolá metodu do_GET().
  8. V metodě do_GET() se vyhodnotí, že cesta je /reservations.
  9. Spustí se logika pro načtení rezervací.
  10. Výsledek se převede na JSON.
  11. Server vrátí odpověď zpět prohlížeči.
  12. Prohlížeč odpověď zobrazí nebo s ní dál pracuje.
Tohle je backend v akci:
adresa sama nic nedělá. Požadavek spustí kód. Kód vyrobí odpověď.
Blok 8: jak Python pozná, co má spustit

V souboru server_basic.py se používá BaseHTTPRequestHandler. To je základní třída, která už ví, jak obsluhovat HTTP požadavky.

Má jednoduché pravidlo: podle typu HTTP metody hledá metodu se správným názvem.

  • když přijde GET, hledá do_GET(),
  • když by přišel POST, hledala by do_POST(),
  • když by přišel DELETE, hledala by do_DELETE().
Místo, kam dorazí GET požadavek 
def do_GET(self):
    ...

Tato metoda je tedy vstupní místo pro zpracování GET požadavku. Neřeší ještě všechen web. Řeší jen situaci: „Přišel GET. Co s ním?“

Blok 9: co je cesta serveru a jak funguje jednoduchý routing

Server nevrací vždy stejnou odpověď. Rozhoduje se podle cesty, která přišla v požadavku.

V našem případě jsou důležité dvě cesty:

  • / = základní kořenová cesta,
  • /reservations = cesta pro seznam rezervací.

V kódu to vypadá takto:

Rozhodování podle cesty 
if self.path == "/reservations":
    ...
elif self.path == "/":
    ...
else:
    ...

Tohle je nejjednodušší možný routing. Routing znamená: mapování adresy na konkrétní logiku.

Co to znamená v praxi

  • adresa / povede na jednu odpověď,
  • adresa /reservations povede na jinou odpověď,
  • jakákoliv jiná adresa dostane chybu 404.
Blok 10: úplně nejmenší server bez dalších témat – rozbor řádek po řádku

Teď přijde úplně nejmenší možná ukázka serveru. Smysl této ukázky není postavit hotovou aplikaci, ale opravdu přesně pochopit, co dělá každý řádek a jak se z obyčejného Python souboru stane program, který odpovídá přes web.

Nejmenší serverová ukázka 
from http.server import BaseHTTPRequestHandler, HTTPServer
import json


class MyHandler(BaseHTTPRequestHandler):
    def do_GET(self):
        if self.path == "/":
            data = {"message": "Ahoj ze serveru"}
            body = json.dumps(data, ensure_ascii=False).encode("utf-8")

            self.send_response(200)
            self.send_header("Content-Type", "application/json; charset=utf-8")
            self.send_header("Content-Length", str(len(body)))
            self.end_headers()
            self.wfile.write(body)
        else:
            self.send_response(404)
            self.end_headers()


server = HTTPServer(("127.0.0.1", 8000), MyHandler)
print("Server běží na adrese: http://127.0.0.1:8000")
server.serve_forever()

Rozbor opravdu řádek po řádku

Řádek 1: import tříd pro HTTP server 
from http.server import BaseHTTPRequestHandler, HTTPServer

Tímto řádkem se ze standardní knihovny Pythonu načtou dvě třídy:

  • BaseHTTPRequestHandler = základ pro třídu, která bude umět zpracovat HTTP požadavek.
  • HTTPServer = samotný server, který poběží na adrese a portu.

Jinými slovy: první část říká, jak se bude požadavek zpracovávat, druhá část říká, co bude fyzicky naslouchat na síti.

Řádek 2: import modulu json 
import json

Tento řádek načte modul json. Ten budeme potřebovat proto, že odpověď nechceme vracet jako Python slovník, ale jako JSON text.

Python slovník: 

{"message": "Ahoj ze serveru"}

se díky modulu json převede na text, který umí číst prohlížeč nebo JavaScript na stránce.

Prázdné řádky: proč tam jsou

Prázdné řádky nic nespouštějí. Slouží jen pro přehlednost.

V Pythonu se často oddělují:

  • importy,
  • definice tříd a funkcí,
  • hlavní spuštění programu.

Díky tomu je kód čitelnější.

Řádek 4: definice třídy MyHandler 
class MyHandler(BaseHTTPRequestHandler):

Tady vzniká vlastní třída MyHandler.

Závorka s BaseHTTPRequestHandler znamená, že tato třída z ní dědí. Tedy: přebírá její základní chování a může si ho upravit nebo doplnit.

V praxi to znamená: Python už má připravený základ pro zpracování HTTP požadavků a my do něj jen dopíšeme, co se má stát při konkrétní metodě, například při GET.

Řádek 5: definice metody do_GET 
def do_GET(self):

Tohle je jeden z nejdůležitějších řádků celé ukázky.

Název do_GET není vymyšlený náhodně. Je to pravidlo. Když server dostane HTTP požadavek metodou GET, automaticky hledá metodu se jménem do_GET.

Parametr self znamená, že tato metoda pracuje s konkrétní instancí handleru, tedy s konkrétním zpracováním jednoho požadavku.

Přesně řečeno:
Když přijde GET požadavek, Python zavolá právě tuto metodu.
Řádek 6: kontrola cesty 
if self.path == "/":

Tady se kontroluje, na jakou adresu přišel požadavek.

self.path obsahuje cestu z URL. Například:

  • pro http://127.0.0.1:8000/ bude self.path rovno "/",
  • pro http://127.0.0.1:8000/reservations bude self.path rovno "/reservations".

Tento řádek tedy říká: „Pokud přišla kořenová adresa, proveď následující blok kódu.“

Řádek 7: vytvoření Python slovníku s daty 
data = {"message": "Ahoj ze serveru"}

Tady vznikne obyčejný Python slovník.

Zatím to ještě není HTTP odpověď. Zatím to není ani JSON text. Je to jen datová struktura uvnitř Pythonu.

Klíč "message" je název položky a hodnota "Ahoj ze serveru" je text, který se má vrátit.

Řádek 8: převod slovníku na JSON a potom na bajty 
body = json.dumps(data, ensure_ascii=False).encode("utf-8")

Tento řádek dělá dvě různé věci najednou.

První část
json.dumps(data, ensure_ascii=False)

převede Python slovník na JSON text.

Například: 

{"message": "Ahoj ze serveru"}

se převede na textovou JSON reprezentaci.

Parametr ensure_ascii=False říká, že se české znaky nemají zbytečně převádět na escape sekvence a mohou zůstat normálně čitelné.

Druhá část
.encode("utf-8")

převede text na bajty. To je potřeba proto, že síťová komunikace nakonec pracuje s bajty, ne s běžným Python stringem.

Proměnná body tedy obsahuje tělo odpovědi připravené k odeslání.

Řádek 10: odeslání stavového kódu 200 
self.send_response(200)

Tímto řádkem server říká: „Požadavek byl zpracován úspěšně.“

Kód 200 znamená OK. Je to základní HTTP stav pro úspěšnou odpověď.

Řádek 11: hlavička Content-Type 
self.send_header("Content-Type", "application/json; charset=utf-8")

Tento řádek posílá hlavičku odpovědi.

Hlavička Content-Type říká klientovi, jaký typ obsahu přichází.

  • application/json = vracený obsah je JSON,
  • charset=utf-8 = text je kódovaný jako UTF-8.

Díky tomu prohlížeč nebo JavaScript ví, jak s odpovědí pracovat.

Řádek 12: hlavička Content-Length 
self.send_header("Content-Length", str(len(body)))

Tady server říká, jak dlouhé bude tělo odpovědi.

len(body) spočítá počet bajtů v proměnné body. Protože hlavičky posílají text, převede se délka ještě přes str(...) na řetězec.

Tahle hlavička není v malých ukázkách vždy nutná, ale je správné ji ukázat, protože patří k poctivému HTTP přístupu.

Řádek 13: konec hlaviček 
self.end_headers()

Tímto řádkem server říká: „Hlavičky jsou hotové, teď začne tělo odpovědi.“

Dokud se nezavolá end_headers(), server ještě pořád skládá hlavičky.

Řádek 14: odeslání samotných dat 
self.wfile.write(body)

Tady se konečně odešle skutečný obsah odpovědi.

self.wfile je výstupní proud, přes který handler zapisuje data klientovi.

Metoda write(body) pošle bajty uložené v proměnné body.

Řádek 15–17: co se stane při jiné cestě 
else:
    self.send_response(404)
    self.end_headers()

Pokud adresa není /, provede se větev else.

Server pak vrátí stav 404, tedy: „Taková cesta na tomto serveru neexistuje.“

Tělo odpovědi se tu neposílá, vrací se jen samotný stav.

Řádek 20: vytvoření serveru 
server = HTTPServer(("127.0.0.1", 8000), MyHandler)

Tímto řádkem se vytvoří instance serveru.

Do serveru se předávají dvě důležité věci:

  • ("127.0.0.1", 8000) = adresa a port, na kterých má server běžet,
  • MyHandler = třída, která říká, jak se mají požadavky zpracovávat.

Jinými slovy: server teď ví, kde má naslouchat a komu má předat přijatý požadavek.

Řádek 21: výpis informace do konzole 
print("Server běží na adrese: http://127.0.0.1:8000")

Tento řádek nic neposílá do prohlížeče. Jen vypíše informaci do konzole, aby bylo jasné, kde server běží.

Je to užitečné hlavně při spuštění, protože hned vidíš, jakou adresu otevřít v prohlížeči.

Řádek 22: spuštění serverové smyčky 
server.serve_forever()

Tohle je řádek, který celý server opravdu rozběhne.

Metoda serve_forever() znamená: server bude pořád běžet, bude čekat na další a další požadavky a nebude se po prvním požadavku ukončovat.

Bez tohoto řádku by se server sice vytvořil, ale hned by skončil.

Tady se z definice stává běžící proces:
do této chvíle se server jen nastavoval, tímto řádkem se skutečně spustí.

Celá logika této malé ukázky v jedné větě

Server běží na adrese 127.0.0.1:8000, čeká na GET požadavek, při cestě / vrátí JSON zprávu a při jiné cestě vrátí 404.

  • Na adrese / vrací krátkou informaci, že server běží.
  • Na adrese /reservations vrací seznam rezervací jako JSON.

Blok 12: kde se v tom řetězci objeví databáze

Databáze v této kapitole není hlavní téma. Je ale dobré přesně vidět, kde v celém toku stojí.

Databáze je jen jedna část z delšího řetězce:

Celý tok komunikace 
prohlížeč → server → backendová logika → data → odpověď → prohlížeč

Databáze je tedy uvnitř systému, ne na jeho začátku. Prohlížeč nemluví přímo s databází. Prohlížeč mluví se serverem. Server spustí backendovou logiku a ta si teprve případně sáhne na data.

Tím se vysvětluje i důležitý princip:
databáze není webová adresa. Webová adresa spouští kód, a ten se případně obrátí na databázi.
Blok 13: proč server nemá databázi sám zakládat

Tady už se databáze znovu nerozebírá do hloubky, ale je potřeba říct jeden důležitý architektonický princip:

Server nemá při spuštění tiše připravovat celý projekt. Server má hlavně obsluhovat požadavky.

Kdyby server při spuštění rovnou vytvářel novou databázi, začaly by se míchat role:

  • server by zároveň obsluhoval web,
  • a zároveň by tajně připravoval datovou vrstvu.

Čistší je mít to rozdělené:

  • app.py připraví databázi a data, :contentReference[oaicite:4]{index=4}
  • server_basic.py data zpřístupní přes web. :contentReference[oaicite:5]{index=5}

Tím je jasné, kdo za co zodpovídá.

Blok 14: co znamenají odpovědi 200, 404 a 500

Server nevrací jen samotná data. Vrací také stavovou informaci, která říká, jak dopadlo zpracování požadavku.

  • 200 = vše proběhlo v pořádku.
  • 404 = požadovaná cesta na serveru neexistuje.
  • 500 = uvnitř serveru nastala chyba.

Příklady

  • GET / může vrátit 200 a zprávu, že server běží.
  • GET /reservations může vrátit 200 a JSON se seznamem rezervací.
  • GET /neco-co-neexistuje vrátí 404.
  • Pokud se při načítání dat něco pokazí, může přijít 500.
Blok 15: proč to celé dává smysl ještě před FastAPI

Právě tahle čistá verze bez frameworku ukazuje, co je na backendu opravdu podstatné.

  • existuje server,
  • server naslouchá,
  • přijde požadavek,
  • spustí se správná část kódu,
  • vznikne odpověď,
  • odpověď se vrátí klientovi.

Až když je tenhle princip jasný, začne mít FastAPI skutečný smysl. Ne jako magie, ale jako pohodlnější a kratší způsob, jak ten samý princip zapsat.

Přesně řečeno:
FastAPI nemění podstatu backendu. FastAPI jen dává lepší nástroje pro zápis stejné myšlenky.

Co je po této kapitole opravdu pevné

  • Backend je vrstva logiky, ne název frameworku.
  • Server je běžící program, který přijímá požadavky a vrací odpovědi.
  • Komunikace na webu stojí na modelu požadavek → zpracování → odpověď.
  • HTTP je společný jazyk mezi klientem a serverem.
  • GET znamená požadavek na získání výsledku nebo dat.
  • Metoda do_GET() je v Pythonu místo, kam GET požadavek dopadne.
  • Cesta jako / nebo /reservations určuje, jaká logika se spustí.
  • Server vrací nejen data, ale i stavové kódy jako 200, 404 a 500.
